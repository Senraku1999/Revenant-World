## 零、速查表

### 日常

```

```

### 审查

```

CB 全部      → 全项目 CB 扫描
同步          → 中英文 JSON 语义一致性
同步 角色     → 指定角色 JSON 对比（如 同步 薰）
逻辑 角色     → 开场白六项逻辑核查（如 逻辑 薰）
逻辑 全部     → 全项目开场白核查

```


## 二、工具详解

### 2.1 三个 Workflow（日常主力）

#### `/quick-check` — 增量审查

**触发：** `检查` / `提交`

**做什么：**
1. `git diff` 检测变更文件
2. 运行自动检查脚本（JSON 语法 / Token / Em dash / 枚举值 / 评级一致性）
3. 对变更的 MD 做标点审查（六条硬规则）
4. 对变更的 MD 做 ControlBoundary 审查
5. T0/T1/T2 分级输出

**规模：** 4 agent / ~96K token / ~3 分钟（有缓存更快）

**对比全量：** 节省 74% token，跳过未变更的 80+ 文件。

---

#### `/change-impact` — 变更影响分析

**触发：** `影响`（auto） / `影响 关键词`（手动）

**做什么：**
1. Grep 扫描全项目引用
2. 按文件类型分为三级：
   - **MUST_SYNC** — 定义性文件（world info / 时间线 / JSON / 关系网），必须同步
   - **SHOULD_CHECK** — 叙事文件（简介 / 开场白 / 小说化内容），建议检查
   - **NO_ACTION** — 工具文件（审查文件 / .claude），无需改动
3. 输出每个文件的具体引用位置和修复顺序

**规模：** 3 agent / ~49K token / ~30 秒

---

#### `/full-review` — 全量审查

**触发：** `全量`

**做什么：** 六阶段完整流程——
1. 自动脚本扫描
2. 标点审查（全量 93 个 MD）
3. 逻辑审查（跨文件交叉比对）
4. 学术五视角并行审查
5. 汇总分级（T0/T1/T2）
6. 清理中间产物

**规模：** 11 agent / ~370K token / ~8 分钟

---

### 2.2 三个审查 Agent（专项深入）

#### `boundary-reviewer` — ControlBoundary 审查

**触发：** `CB` / `CB 全部`

**检查规则：** 允许试探发问，禁止明确设定玩家身份、目的或关系。

**违规示例：**
- "你是刚入职的狩灵"（身份设定）
- "你来这里是为了杀我"（目的设定）
- "你是我的搭档"（关系设定）

**不违规：** "你找哪位？""来委托的？""你是协会的人？"（试探性发问）

---

#### `json-sync-reviewer` — 中英文 JSON 一致性

**触发：** `同步` / `同步 角色名`

**检查内容：**
- 结构层：键集合一致、数组长度一致、嵌套层级一致
- 语义层：描述字段信息等量、性格字段语气一致、能力字段格式一致
- 关系层：关系键中英文对应、关系描述不增不减
- 对话层：对话示例数量一致、翻译准确

---

#### `opening-logic-reviewer` — 开场白六项核查

**触发：** `逻辑 角色名` / `逻辑 全部`

**六项机械核查：**

| # | 核查项 | 检查什么 |
|---|--------|----------|
| 1 | 肢体逻辑 | 左右不混淆、视角不矛盾、动作物理可能 |
| 2 | 能力一致性 | 展示的能力在 JSON 中有定义、不超上限、触发条件一致 |
| 3 | 武器一致性 | 名称/外观/特性与 JSON 一致、weapon=None 不出武器 |
| 4 | 时间一致性 | 开场白时间与 char_background 事件年份吻合 |
| 5 | 空间逻辑 | 场景变动有过渡、位置感知合理、物品位置前后一致 |
| 6 | 动作闭环 | 受伤有过程、物品出现有交代、能力触发有条件 |

## 三、典型工作流

### 场景 A：日常修改 → 提交

```
改文件 → 检查 → 修（如有问题）→ 提交
```

实际只需说三个词。

### 场景 B：修改共享事件

```
影响 事件名 → 按 MUST_SYNC 清单逐文件改 → 检查 → 修 → 提交
```

### 场景 C：写新角色卡

```
写完四个文件 → 检查 → CB → 同步 → 逻辑 角色名 → 修 → 提交
```

### 场景 D：版本发布

```
检查 → 全量 → 按报告逐级修复 → 检查 → 提交
```

## 五、补充

### 与传统命令对照

| 旧方式 | 新方式 |
|--------|--------|
| `python "创作者文件/审查文件/自动检查.py"` + 人眼读输出 | `检查` |
| 手动 grep 搜索所有引用 | `影响 关键词` |
| 人眼逐行查 CB 违规 | `CB` |
| 人眼对比中英文 JSON | `同步` |
| `/full-review` | `全量` |

### 配置文件位置

```
.claude/
├── CHEATSHEET.md              # 速查表（最短参考）
├── TOOLS_GUIDE.md             # 详细使用指南
├── agents/                    # 13 个 agent 定义
├── workflows/                 # 3 个 workflow 脚本
└── settings.local.json        # 权限配置
```

### 注意事项

- 所有 agent 和 workflow 仅在 Claude Code 会话中生效
- 首次运行无缓存时较慢，后续运行有缓存加速
- 如有新角色类型或新文件格式，可能需要更新 agent 规则
- 遇到误报或漏报，直接描述问题即可，pm-orchestrator 会调整
