# CLAUDE.md

## 项目概述

狩灵世界观是以当代日本双重社会为背景的灵能题材角色扮演世界观。每个角色含三个文件：`{角色名}.json`（角色卡）、`{角色名}简介.md`、`{角色名}开场白.md`。

| 目录 | 内容 |
| ------ | ------ |
| `创作者文件/创作文件/` | 创作指导、格式标准 |
| `创作者文件/审查文件/` | 审查规则、标点国标、自动检查脚本 |
| `创作者文件/导出文件/world info/` | 世界观与规则条目（World Info） |
| `创作者文件/导出文件/关系网/` | 按阵营分目录的角色关系副本 |
| `创作者文件/导出文件/世界书/` | 按阵营分册的 ST 世界书 JSON |
| `角色卡/` | 全部角色卡（按阵营分目录） |
| `导出角色卡/` | 导出角色卡 |
| `小说化内容/` | 世界观叙事文学 |

## 一、核心规则

**以卡为准**：规范定义字段和格式，卡片定义实际写法。创作或修改字段值时，必须确认至少三张同类角色卡的同字段写作方式。规范与全体卡片不一致时，以卡片为准。

**标点**：中文遵循 GB/T 15834，数字用法遵循 GB/T 15835，异形词遵循 GF 1001—2001。英文遵循英文标准语法。

- 角色卡（JSON/简介/开场白）：叙述仅 `，` `。` `、` 三标点，`""` 为唯一合法引号，`；` 仅限并列关系，`——` `……` 仅限对话内。嵌套引号（`""` 内 `''`）合法。
- 小说化内容：中国标准文学标点，中文引号 `""`，句尾 `？！` 解锁，`——` 和 `……` 在叙述中同样禁止（仅限对话内）。
- `：` 在枚举（`X：A、B、C`）、解释（`X：Y说明`）、引述（`说："..."`）中均为合法用法。

标点审查三级体系：一级 GB/T 15834 → 二级三步法 → 三级项目追加，详见 `创作者文件/审查文件/`。

**JSON**：遵循 RFC 8259。UTF-8 编码，无 BOM。中英文 JSON 同步修改，结构一致。

**ControlBoundary**：允许试探发问，禁止明确设定玩家身份、目的或关系。

**共享事件同步**：修改涉及多个角色的共享事件后，必须逐文件检查全部关联角色，确认时间锚点、责任归属、基本事实无矛盾。

## 二、创作

> **每次修改角色文件之前，以下两步不得跳过：**

**第一步：重读规范（禁止依赖会话记忆）。** 根据修改类型，用 Read 打开对应指导文件：

| 修改内容 | 必读文件 |
| ------ | ------ |
| JSON | `创作者文件/创作文件/狩灵角色卡创作指导.md`、`创作者文件/创作文件/角色卡标准格式.md` |
| 简介 | `创作者文件/创作文件/狩灵简介创作指导.md`、`创作者文件/创作文件/简介标准格式.md` |
| 开场白 | `创作者文件/创作文件/狩灵开场白创作指导.md` |
| 任何角色文件 | `创作者文件/导出文件/world info/宏观规则.md` |
| 导出/PNG/世界书/关系网 | `创作者文件/导出文件/角色卡导出规范.md` |
| appearance 字段 | `创作者文件/创作文件/Appearance字段格式指导.md` |
| 术语/专有名词 | `创作者文件/创作文件/专有名词全称表.md` |
| 基础能力字段 | `创作者文件/创作文件/基础能力等级示例.md` |

**第二步：修改后跑检查，全部达标再报告。** 运行 `npx tsx "创作者文件/审查文件/自动检查.ts"`（命令见第四节）。任一检查未通过 → 继续修改 → 再跑检查 → 直到全部达标。不等用户提醒。

## 三、团队

Agent 配置文件位于 `.claude/agents/`。

### 项目经理

| Agent | 职责 |
| ------ | ------ |
| `pm-orchestrator` | 任务分解、agent 调度、进度追踪、结果汇总 |
| `project-shepherd` | 跨职能协调、时间线管理、资源与风险管控 |

### 学术团队

| Agent | 领域 |
| ------ | ------ |
| `anthropologist` | 文化系统、亲属关系、仪式、帮会亚文化、双重社会 |
| `geographer` | 城市分布、地盘空间、场景地理锚定、通勤逻辑 |
| `historian` | 时间线、事件因果、年代断代、组织起源 |
| `narratologist` | 叙事结构、开场白三段式、对话风格、角色 voice |
| `psychologist` | 人格张力、创伤处理、动机可信度、缺陷真实性 |

### 审查团队

| Agent | 职责 |
| ------ | ------ |
| `review-coordinator` | 流程管理、任务分配、结果汇总、最终裁决 |
| `punctuation-reviewer` | 按 GB/T 15834 + 项目规则，逐字标点扫描 |
| `logic-reviewer` | 跨文件/跨角色逻辑矛盾检测 |
| `boundary-reviewer` | ControlBoundary 边界审查 |
| `json-sync-reviewer` | 中英文 JSON 结构一致性 |
| `opening-logic-reviewer` | 开场白六项机械核查 |

全量审查（`full-review`）：全角色覆盖，无抽样。自动脚本 → 标点+逻辑并行 → 全角色扫描 → 学术五视角并行 → 汇总分级。

抽样审查（`sampling-review`）：轻量快速。自动脚本 → 标点+逻辑并行 → 统一抽样（8 角色）→ 学术五视角并行 → 汇总分级（含未抽样风险提示）。

> **审查后清理**：审查结束后删除所有临时文件（`*_output.txt`、`*_extract*.ts`、`cross_file_*`、根目录下以 `_` 开头的 `.ts`/`.txt` 等）。永久脚本为 `自动检查.ts`、`一级审查.ts`、`二级审查.ts`、`全量导出.ts`，统一使用 `npx tsx` 运行。

## 四、常用命令

```bash
# 审查
npx tsx "创作者文件/审查文件/自动检查.ts"               # 全项目自动扫描
npx tsx "创作者文件/审查文件/一级审查/一级审查.ts"       # 一级国标标点审查
npx tsx "创作者文件/审查文件/二级审查/二级审查.ts"       # 二级三步法机械预筛
# 导出管线（按序运行）
npx tsx "创作者文件/导出文件/导出关系网.ts"              # 管线一：角色卡 → 关系网
npx tsx "创作者文件/导出文件/导出世界书.ts"              # 管线二：关系网 → 世界书
npx tsx "创作者文件/导出文件/导出角色卡.ts"              # 管线三：底图 + 世界书 → PNG
npm run 全管线                                           # 一键跑全管线
# 工具
node -e "JSON.parse(require('fs').readFileSync('文件.json','utf-8'))" > /dev/null
node -e "const f=require('fs');console.log(f.readdirSync('目录').length)"
# 内联脚本规则（本机 npx tsx -e 被 Windows npx.cmd 换行截断，禁止使用，特例也已被 settings 阻断）
node -e "内联纯 JS 脚本"                                       # 首选：CJS 包（js-tiktoken 等）直接 require
node node_modules/tsx/dist/cli.mjs -e "内联 TS 脚本"           # 需要导入项目 TS 模块时使用（绕开 npx.cmd，多行安全）
# 临时 TS 文件也是合法替代：Write → npx tsx 文件.ts → rm
```
