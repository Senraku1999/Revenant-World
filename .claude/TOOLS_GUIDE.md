## 一、快速总览

### 三个技能（日常最常用）

| 技能 | 触发方式 | 耗时 | 用途 |
|------|----------|------|------|
| `/quick-check` | 直接输入 | ~3 分钟 | 提交前检查：自动脚本 + 标点 + ControlBoundary |
| `/change-impact 关键字` | 直接输入 | ~30 秒 | 改共享实体前，查影响范围 |
| `/full-review` | 直接输入 | ~8 分钟 | 版本发布前全量审查 |

### 三个审查 agent（可单独调用）

| Agent | 触发方式 | 用途 |
|-------|----------|------|
| `boundary-reviewer` | "检查 XX 开场白的 CB""跑一下边界审查" | ControlBoundary 违规检测 |
| `json-sync-reviewer` | "对比 XX 的中英文 JSON""检查 JSON 同步" | 中英文 JSON 语义一致性 |
| `opening-logic-reviewer` | "核查 XX 开场白逻辑""跑六项核查" | 开场白肢体/能力/武器/时间/空间/闭环 |

## 二、三个技能详解

### 2.1 `/quick-check` — 提交前增量审查

**什么时候用：** 每次修改文件后、提交 git 之前。

**它做什么：**
1. 自动检测 git diff 中的变更文件
2. 运行 `审查脚本_自动检查.py`（JSON 语法、Token、Em dash、枚举值等）
3. 对变更的 MD 文件做标点审查（六条硬规则）
4. 对变更的 MD 文件做 ControlBoundary 审查（身份/目的/关系预设）
5. 汇总输出 T0/T1/T2 分级报告

**典型用法：**
```
/quick-check
```

**输出示例：**
```
总评：发现 4 个问题（1 个 T0，3 个 T1）

T0：写乐 D 段 Token 404，超出上限 400
T1：静流/零/克羽 关系键中英文不匹配
标点零问题，CB 零问题
```

**对比全量审查：** 4 agent vs 11 agent，96K token vs 370K token，节省 74%。

---

### 2.2 `/change-impact` — 变更影响分析

**什么时候用：** 修改共享实体（事件名、角色名、组织名、时间线日期等）之前，想知道会影响哪些文件。

**两种模式：**

**关键字模式（明确知道改了哪个实体）：**
```
/change-impact 第一次扭曲
/change-impact 花坂 枫
/change-impact 灵力回路
```

**auto 模式（不确定影响了什么，让系统自动从 git diff 提取）：**
```
/change-impact
```
不加任何参数即为 auto 模式。系统从当前变更文件中自动提取 1-3 个最重要的实体名，逐个分析影响范围。

**它做什么：**
1. 用 Grep 扫描全项目引用该关键字的所有文件
2. 按文件类型分为三级：
   - **MUST_SYNC**：定义性文件（world info、时间线、JSON、关系网）
   - **SHOULD_CHECK**：叙事文件（简介、开场白、小说化内容）
   - **NO_ACTION**：工具文件（审查文件、创作者文件、.claude）
3. 输出每个文件的具体引用位置和修复顺序建议

**输出示例：**
```
修改「第一次扭曲」影响 14 个文件，7 个 MUST_SYNC，6 个 SHOULD_CHECK

MUST_SYNC（优先修复）：
  事件卡/第一次扭曲/第一次扭曲.json — 事件卡定义
  协会狩灵角色卡/薰/薰_zh.json — origin 字段
  ...
修复顺序：事件定义 → 角色卡 → 时间线 → 叙事文件
```

---

### 2.3 `/full-review` — 版本发布前全量审查

**什么时候用：** 准备发布新版本、大量修改后需要全面检查、一个月跑一次。

**它做什么：** 完整六阶段流程——
1. 自动脚本扫描
2. 标点审查（全量 MD 文件）
3. 逻辑审查（跨文件交叉比对）
4. 学术五视角并行审查（人类学/地理/历史/叙事/心理）
5. 汇总分级（T0/T1/T2）
6. 清理中间产物

**典型用法：**
```
/full-review
```
或直接说"全量审查"。

## 三、三个审查 Agent 详解

这些 agent 可以单独调用，只需在对话中描述需求，项目经理会自动调度。

### 3.1 `boundary-reviewer` — ControlBoundary 审查

**检查什么：**
- 是否明确设定了玩家的身份（"你是个狩灵新手"）
- 是否明确设定了玩家的目的（"你来这里是为了杀我"）
- 是否明确设定了玩家与角色的关系（"你是我的搭档"）
- 对话示例是否预设了玩家回应
- 是否出现连续 ≥3 个独立发问句（问题循环风险）

**不报什么（不要误判）：**
- 试探性发问（"你找哪位？""来委托的？""你是协会的人？"）— 不违规

**典型用法：**
```
检查一下薰开场白的 ControlBoundary
审查所有新创建的开场白 CB
跑一下全项目 CB 扫描
```

---

### 3.2 `json-sync-reviewer` — 中英文 JSON 一致性

**检查什么：**
- 结构层：两文件键集合一致、数组长度一致、嵌套层级一致
- 语义层：描述字段信息等量（无遗漏无添加）、性格字段语气一致、能力字段格式一致
- 关系层：关系键中英文对应、关系描述信息等量
- 对话层：对话示例数量一致、翻译准确且口吻一致

**典型用法：**
```
对比一下薰的中英文 JSON
检查所有角色卡的中英文同步情况
审查静流的 JSON 翻译一致性
```

---

### 3.3 `opening-logic-reviewer` — 开场白六项逻辑核查

**六项机械核查：**
1. **肢体逻辑**：左右不混淆、视角不矛盾、不超过人体工学极限
2. **能力一致性**：展示的能力在 JSON 中有定义、不超上限、触发条件一致
3. **武器一致性**：名称/外观/特性与 JSON 一致、weapon=None 不出武器
4. **时间一致性**：开场白事件与 char_background 时间线吻合
5. **空间逻辑**：场景变动有过渡、位置感知合理、物品位置前后一致
6. **动作闭环**：受伤有过程、物品出现有交代、能力触发有条件

**典型用法：**
```
核查一下薰开场白的逻辑
用六项核查检查所有新开场白
审查菲利普开场白的武器和能力一致性
```

## 四、日常工作流

### 场景 A：写一张新角色卡

```
1. 写完四个文件后 → /quick-check
2. 单独检查 CB → "检查 XX 开场白的 ControlBoundary"
3. 单独检查中英文同步 → "对比 XX 的中英文 JSON"
4. 核查开场白逻辑 → "跑一下 XX 的六项核查"
5. 全部通过 → git commit
```

### 场景 B：修改共享事件（如改名、改时间）

```
1. 先跑影响分析 → /change-impact 第一次扭曲
2. 按 MUST_SYNC 清单逐文件修改
3. 修改完成后 → /quick-check
4. 全部通过 → git commit
```

### 场景 C：不确定改了什么会影响什么

```
1. 改完文件后 → /change-impact（auto 模式）
2. 查看自动提取的实体和影响范围
3. 如有遗漏，再用关键字模式补查 → /change-impact 关键字
4. 按影响清单同步
5. → /quick-check
```

### 场景 D：版本发布前

```
1. /quick-check（确保没有低级错误）
2. /full-review（全面检查）
3. 按 T0 → T1 → T2 修复
4. 修复后重新 /quick-check
5. → git commit + push
```

## 五、与项目经理配合

所有 agent 都可以通过项目经理（pm-orchestrator）调度。你不需要记住每个 agent 的名字，只需要描述需求：

```
"帮我全面审查一下最近改的文件"          → 自动调度 quick-check
"我改了时间线，看看影响了哪些角色"       → 自动调度 change-impact
"检查新角色卡有没有违规"                → 自动调度 boundary-reviewer + opening-logic-reviewer
"准备发布，跑全量审查"                  → 自动调度 full-review
```

## 六、常见问题

**Q: quick-check 和 full-review 有什么区别？**
A: quick-check 只检查 git diff 中变更的文件，3-4 个 agent，适合日常频繁使用。full-review 扫描全部 93 个 MD + 所有 JSON，11 个 agent，适合版本发布前。

**Q: 为什么 change-impact 有时返回"未检测到实体"？**
A: auto 模式从 git diff 中提取实体名。如果你只改了措辞、标点、格式，没有改动角色名/事件名/组织名等专有名词，就会返回空。此时不需要影响分析。

**Q: 三个审查 agent 可以同时跑吗？**
A: 可以。直接说"全面审查 XX 角色卡"，项目经理会并行调度。

**Q: 需要记住所有 trigger 关键词吗？**
A: 不需要。直接描述你要做什么，项目经理会自动匹配最合适的 agent/workflow。
