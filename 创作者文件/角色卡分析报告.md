## 三、角色字段对比分析

### 3.1 Overview 质量分布

Overview 是角色卡的"一句话核心"，应为对角色本质的精炼概括。实际执行中存在显著差异：

**优秀示例（一句话锁定角色核心）**：
- 巴拉尔："Burns at quintuple rate, knows her remaining years."（暴食回路的代价，言简意骇）
- 诺克斯："Infiltration and assassination specialist whose unsettling presence keeps even colleagues at a distance."（双面性：能力+性格影响）
- 静流："Former 狩灵协会2科 member who retired after her partner died shielding her from a 扭曲, now running a solo Tokyo office."（完整叙事弧线）
- 劫："Former 狩灵协会4科 狩灵, labeled a 罪灵 after resigning to hunt 烬."（身份→动机→状态，一气呵成）

**偏短示例（缺乏信息密度）**：
- 薰："Guardian of 天丛云剑, confined to 狩灵协会 grounds."——未揭示其"弑父封剑"的核心叙事
- 贝拉："Faithfully completes every task with little emotion."——未提及其与贝尔金的师徒关系
- 红："Spent fifteen years as a 游魂 in the American Southwest before returning to Japan."——更接近 origin 叙事而非 overview

**规律**：Overview 理想长度约 2-3 句，需同时涵盖"身份定位 + 核心矛盾/特质 + 当前状态"三个维度。当前约 40% 的角色 overview 偏向单一维度。

### 3.2 关系字段（char_relationships）完整度

8 个角色关系字段为空：巴拉尔、葵、诺克斯、千子、特蕾莎、响、光、莱薇。

其中：
- **合理空缺**（角色设定为独立/孤僻）：特蕾莎（修女，以信仰为伴）、诺克斯（潜入暗杀者，刻意保持距离）、葵（热衷武器试验，社交圈限于工坊）
- **可补充**（角色设定中有潜在关系维度）：巴拉尔（1科副科长，应与科长或其他同僚有工作关系）、响（团队型角色，应与 2 科队友有互动）、光（社区保护者，与被保护者有情感纽带）、莱薇（酒吧老板，与常客有情报交换关系）

规律：关系字段不为空时，平均 3-4 条关系。花坂家族三人平均 4.3 条（含家族成员）。协会干部（科长/副科长）通常有 3-5 条上下级/跨科关系。

## 五、全新审查方法论

基于对全部 45 个角色逐字段通读后发现的规律性问题，设计以下审查流程。此方法论推倒旧有思路，不参考任何旧版审查文档。

### 5.1 核心洞察

经过三天逐角色优化和本次全量通读，我们发现了本项目角色卡质量的真实瓶颈：

1. **自动脚本已经解决 95% 的形式问题**。标点、术语、评级格式、em dash、Token 计数——这些已实现全自动检测。
2. **剩余问题全部集中在语义层**：overview 是否抓住了角色核心、关系字段是否反映角色社交维度、origin 与 overview 是否各司其职、对话是否贴合角色性格。
3. **跨角色一致性在反复迭代中已高度稳定**。本次验证了 10 组跨角色关系链，无矛盾。

因此，新的审查方法论应**将重心从形式检查转向语义审查**。

### 5.2 三步审查流程

#### 第一步：自动脚本（必跑，1 分钟）

```
python "创作者文件/审查脚本_自动检查.py"
```

通过标准：Token 在预算内、JSON 语法无误、枚举值合法、em dash 零命中、MD 标点合规、评级一致。

不通过则修改后重跑，直到全部通过。此步产出纯客观数据。

#### 第二步：字段语义审查（逐角色人工/Agent 通读）

对每个角色，按以下清单逐项检查，以"是/否"作答：

**Overview 三维度检查**：
- 是否揭示了角色的核心身份？
- 是否暗示了核心矛盾或独特之处？
- 是否给出了当前状态/定位？

通过标准：3/3 或至少 2/3 且第三项在后续字段中有充分展开。

**Origin vs Overview 分工检查**：
- origin 是否在讲故事（人生关键转折），overview 是否在给结论（角色本质概括）？
- 两者是否存在信息高度重复？（如红的情况：overview 说"在美国当了十五年游魂"，origin 也说同一件事）

**关系字段必要性质疑**：
- 该角色的设定（性格、职务、社会位置）是否天然蕴含关系？
- 如果是，关系字段中是否反映了这些天然关系？
- 如果否（如终极孤僻者），空缺是否合理？

**对话贴合度抽查**：
- 随机抽取一条 dialogue_example，读完后能否仅凭语气判断这是哪个角色？
- 三条 dialogue_example 是否展现了角色的不同侧面（如：战斗面、日常面、情感面）？

#### 第三步：跨角色关系一致性（修改后必跑）

每次修改涉及跨角色关系时（如在 A 的 relationships 中新增或修改了与 B 的关系），必须：
1. 用 Grep 搜索 B 的 JSON 文件中是否已有对应关系
2. 如果无，同步添加双向关系
3. 检查情感态度是否对等（如 A 认为"信赖"，B 认为"负担"，这种不对等是角色设计还是错误）

自动化辅助：
```bash
grep -r "角色名" --include="*.json" .
```
检查该角色名在所有 JSON 中的出现次数是否与预期一致。

### 5.3 审查频率与范围

| 触发条件 | 审查范围 | 执行步骤 |
|------|------|------|
| 新建角色 | 全量（自动 + 语义 + 跨角色） | 三步全部 |
| 修改已有角色 JSON | 该角色语义 + 受影响的跨角色关系 | 第一步必跑 + 相关部分 |
| 修改已有角色简介/开场白 | 该文件 Token + 标点 | 第一步 |
| 批量调整（如术语统一） | 第一步 + 受影响字段抽查 | 第一步 + 第二步抽查 |
| 定期维护（建议每 10 个角色修改后） | 全量自动 + 跨角色一致性抽样 | 第一步 + 第三步关键链 |

### 5.4 Agent 调度原则

基于本次分析经验，Agent 调度的有效模式为：

- **logic-reviewer**：在跨角色关系修改后调用，做双向一致性验证。不应在初次创作时调用（会增加不必要的往返）。
- **narratologist**：在新角色创作完成后调用，评估 overview 三维度、dialogue 贴合度、开场白三段式结构。这是产出质量的最关键 gate。
- **psychologist**：在角色有复杂心理动机时调用（如罪灵、创伤角色）。本次通读中，静流、劫、零、写乐等角色的心理描写已是高质量水平，psychologist 应在初次创作而非迭代优化时介入。
- **anthropologist / historian / geographer**：仅在涉及文化习俗、历史断代、地理定位的具体问题时调用。基于本次分析，这三者在常规角色审查中并非必需。

核心原则：各 Agent 在第一次创作时介入效果最好，后期迭代优化时主要依赖自动脚本 + 人工通读。

### 5.5 质量门禁

角色卡交付前必须满足：
1. 自动脚本全部通过
2. Overview 三维度至少 2/3
3. 关系字段不为空（除非角色设定有充分理由空缺）
4. 三条 dialogue_example 能从语气区分角色
5. 涉及的任何其他角色的关系字段已同步更新并保持一致
