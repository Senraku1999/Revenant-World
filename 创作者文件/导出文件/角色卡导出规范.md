# 角色卡导出规范

---

## 导出操作强制前置

> **此节优先级高于一切。违反即错误，无例外。**

执行以下任一操作前，第一步必须 Read 本文件：导出角色卡 JSON、导出角色卡 PNG、更新/重建世界书、同步关系网副本。禁止凭记忆执行。

**修改后导出管线**：角色文件修改定档后按顺序执行——① 同步关系网全部副本（`npm run 导出关系网`），② 重建世界书（`npm run 导出世界书`），③ 更新角色卡 PNG（`npm run 导出角色卡`）。

**PNG 导出前置检查**：执行 PNG 导出前，必须先确认 `底图/` 中已存在对应角色的底图 PNG 文件。无底图不导出。

---

## 数据从源文件解析

> **此节优先级等同于导出操作。违反即错误，无例外。**

涉及 20 条以上清单/列表/表格数据的转移操作——如触发词表角色名、世界书分配表、文件目录清单——必须写脚本从源文件解析。禁止手动誊抄。人眼扫 50+ 条目必然遗漏。

## 输出格式

SillyTavern 实际读取的是**扁平结构**——所有核心字段在顶层，同时 `data` 对象内镜像一份。

```json
{
  "name": "狩灵 · {char_name}",
  "description": "{英文 JSON 全文，\\r\\n 换行}",
  "personality": "",
  "scenario": "",
  "first_mes": "{开场白原文，\\r\\n 换行}",
  "mes_example": "",
  "creatorcomment": "{简介原文，\\r\\n 换行}",
  "avatar": "none",
  "chat": "狩灵 · {char_name} - {导出时间戳}",
  "talkativeness": "0.5",
  "fav": false,
  "tags": ["狩灵", "现代", "超自然", "角色扮演", "战斗", "sfw"],
  "spec": "chara_card_v2",
  "spec_version": "2.0",
  "create_date": "{导出时间戳}",
  "data": {
    "name": "{同上}",
    "description": "{同上}",
    "personality": "",
    "scenario": "",
    "first_mes": "{同上}",
    "mes_example": "",
    "creator_notes": "{简介原文，\\r\\n 换行}",
    "system_prompt": "",
    "post_history_instructions": "",
    "tags": ["狩灵", "现代", "超自然", "角色扮演", "战斗", "sfw"],
    "creator": "千乐",
    "character_version": "",
    "alternate_greetings": [],
    "extensions": {
      "talkativeness": "0.5",
      "fav": false,
      "world": "{世界书名}",
      "depth_prompt": {
        "prompt": "",
        "depth": 4,
        "role": "system"
      }
    },
    "character_book": {
      "name": "{世界书名}",
      "entries": "{世界书全部条目}"
    }
  }
}
```

### 关键规则

| 规则 | 说明 |
|------|------|
| `\r\n` | description / first_mes / creatorcomment 内部字符串的换行符，**不是外层 JSON 的换行符** |
| 扁平 + 镜像 | 顶层和 `data` 内同时存在 name/description/first_mes 等字段，值相同 |
| `creatorcomment` vs `creator_notes` | 顶层用 `creatorcomment`，`data` 内用 `creator_notes` |
| `chat` | 格式 `"狩灵 · {char_name} - {M/D/YYYY @HHh MMm SSs SSSms}"` |
| `character_version` | **空字符串** `""`，不是 `"1.0"` |
| `avatar` | 固定 `"none"` |
| `extensions.world` | 世界书文件名（不含路径和扩展名） |

## 世界书文件格式

独立世界书 JSON 文件（`成品世界书/` 下）必须包含顶层 `name` 字段：

```json
{
  "name": "狩灵 {阵营名}",
  "entries": [...]
}
```

> `name` 缺失导致 SillyTavern 世界书列表中文件无名显示，无法识别和管理。

## 世界书条目格式

每条记录必须包含以下全部字段。缺任何一项导致 SillyTavern 导入失败。

```json
{
  "id": 0,
  "keys": [],
  "secondary_keys": [],
  "comment": "条目名",
  "content": "...",
  "constant": true,
  "selective": true,
  "insertion_order": 100,
  "enabled": true,
  "position": "after_char",
  "use_regex": true,
  "extensions": {
    "position": 1,
    "exclude_recursion": false,
    "display_index": 0,
    "probability": 100,
    "useProbability": true,
    "depth": 4,
    "selectiveLogic": 0,
    "group": "",
    "group_override": false,
    "group_weight": 100,
    "prevent_recursion": false,
    "delay_until_recursion": false,
    "scan_depth": null,
    "match_whole_words": null,
    "use_group_scoring": false,
    "case_sensitive": null,
    "automation_id": "",
    "role": 0,
    "vectorized": false,
    "sticky": 0,
    "cooldown": 0,
    "delay": 0
  }
}
```

| 字段 | 值 |
|------|-----|
| `selective` | 始终 `true` |
| `use_regex` | 始终 `true` |
| `enabled` | 始终 `true` |
| `position` | 宏观条目（keys=[]）→ `"after_char"`；角色条目（keys 非空）→ `"before_char"` |
| `secondary_keys` | 始终 `[]` |
| `extensions` | 必须包含上表全部字段，不可为空 `{}` |

## 导出流程

1. 读取 `{角色}.json` → JSON 序列化为字符串 → `description`（内部 `\r\n`）
2. 读取 `{角色}开场白.md` → `first_mes`（内部 `\r\n`）
3. 读取 `{角色}简介.md` → `creatorcomment` / `data.creator_notes`（内部 `\r\n`）
4. 读取角色对应成品世界书 → `data.character_book`（`{"name": "...", "entries": [...]}`）
5. 拼装为上述扁平 + 镜像结构
6. **JSON 序列化必须用 compact 格式**：`json.dumps(card, separators=(',', ':'))`——无缩进、无空格。`\r\n` 仅存在于内层字符串字段（description / first_mes / creatorcomment），外层 JSON 没有任何换行
7. PNG：将 compact JSON base64 编码，嵌入 PNG 的 tEXt/chara chunk（替换已有 chara chunk，或在 IEND 前插入）

## 导出铁律

> 以下规则来自 2026-07-14 全量导出踩坑总结。违反任一条必然导致导出错误。

### 一、文件匹配用 `card_name`（JSON 文件名 stem），禁止用 `char_name`

`char_name` 与文件名不一致的三种情况：

| 情况 | 示例 |
|------|------|
| char_name 是 NPC | `花坂家宴.json` 的 `char_name` 是 "花坂 晴子"，但配套文件叫 `花坂家宴*` |
| char_name 重复 | `菲利普 · 钢翼.json` 的 `char_name` 也是 "菲利普"，但文件名是 `菲利普 · 钢翼` |
| 无 char_name | 世界观卡（新宿站/协会1科/四色音 · 闪耀舞台）不设 `char_name` 字段 |

**正确做法**：`card_name = os.path.splitext(fn)[0]`，用此查找 `{card_name}开场白.md`、`{card_name}简介.md`、`底图/{card_name}.png`。

### 二、全目录导出，不设过滤器

`角色卡/` 下所有子目录全部导出：角色卡、事件卡、世界观卡、异常角色卡。禁止 SKIP_DIRS。

### 三、世界书 JSON 必须有 `name` 字段

独立世界书文件（`成品世界书/*.json`）顶层须含 `"name"`。缺失时 SillyTavern 世界书列表显示空白。`自动检查.ts` 已加入 `checkWorldbookContentFreshness()` 防御。

### 四、`狩灵.json` 仅含世界信息条目

基础世界书只保留 8 条：宏观规则、世界观、灵能者、评级、异常、狩灵协会、锈钟、天丛云剑。零角色词条。角色词条分散在各阵营世界书中。

### 五、ASSIGN 表名字必须与文件系统完全一致

分配表中的角色名必须与实际 JSON 文件名逐字匹配（含 `·` 等特殊字符）。`四色音 · 闪耀舞台` ≠ `四色音 闪耀舞台`。

### 六、世界书分配：card_name 优先，char_name 做 fallback
