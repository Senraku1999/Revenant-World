# 狩灵世界观 · 角色卡审查执行计划

> 适用于 AI 执行，分六步逐层审查。执行前须先读取规则文件。

---

> 方法论详见 `宏观审查方法论.md`。核心原则：先聚合再比对，禁止逐文件判断跨文件一致性。
>
> **审查优先级**：先按标准中文语法（GB/T 15834）审查（第一级，不可突破），再按个人标点符号规范审查（第二级，在标准语法允许范围内做更严格选择）。详见 `CLAUDE.md` 审查优先级章节。

## 已确认的关键判定边界（无需再次询问）

以下边界已在历次审查中多次确认，直接作为判断依据：

1. **开场白与角色卡 Token 计量**：使用 tiktoken `cl100k_base` 编码计数。**角色卡 JSON 采用四段式分部预算**：A 外壳（300-500）、B+C 能力+关系（0-500）、D 对话（200-400），总计 500-1,400。**开场白**维持 1,000 ±100 token（900-1,100）。计量时以 token 数为准。
2. **色彩阶评级格式**：JSON 中 `色彩阶`、简介中 `色彩阶-称号`，均需带连字符 `-`。检索时匹配带连字符的完整形式。
3. **ControlBoundary 与模糊预设的边界**：试探性发问（"你找哪位？""一路跟着我，是好奇？"）允许；**只要没有明确说明玩家的具体身份、目的或关系，就不算违规**。明确设定的典型违规示例："你是个狩灵新手""你来这里是为了杀我""你是我的搭档"。
4. **评估方对接科室**：协会官方下属事务所所长的评估方为对接科室，可从角色卡 JSON 的 `char_faction` 中反推验证。
5. **special_abilities 字段归类**：带 `灵力回路 -` 前缀的为灵力回路，无此前缀的为特殊能力（如花坂血统）。二者均属于 `char_special_abilities` 字段，仅当角色确无任何回路且无任何特殊能力时，才省略整个字段。
6. **觉醒年龄边界**：觉醒发生在青春期（世界观设定），低于 18 岁的觉醒年龄可接受。工作/从业年龄按现实法律规定须 ≥ 18 岁。
7. **连续问句边界**：开场白中连续 2 个问句可接受，连续 ≥ 3 个独立发问句才构成"问题循环风险"。
8. **一般称呼判定依据**：基于 JSON 的 `char_name` + `char_alias`，非 `char_fullname`。`char_alias` 为字符串 `"None"` 时表示无别名。
9. **协会狩灵评估方**：统一为 `狩灵协会1科`。1科为总务科，负责评级与审查，所有协会狩灵（含科室长）均由 1科 评估。
10. **简介空行规则**：文件开头保留一个空行、综合评估正文与评估方之间保留一个空行，均为格式要求（`简介标准格式.md` §1、§6），非错误。
11. **世界观卡与事件卡审查标准**：不强制遵循角色卡标准格式（五段式、token预算等），但必须严格审查：JSON语法合法性、标点符号合规、叙事流畅性（ControlBoundary、第三人称等）。
12. **关系条目质量审查**：逐对检查双向关系的对称性（两端均有实质性内容，拒绝一端详细一端空洞）、共享事件的多视角一致性（同一事件在不同角色的关系描述中应有角度差异而非复制粘贴）、文学化表达预警（`like halves of` `where words fail` `wears a familiar face` 等降低 AI 可读性的比喻）。
13. **Em dash 自动化扫描**：JSON 非对话字段中 `—`（U+2014）为高频违禁字符，审查脚本 `审查脚本_自动检查.py` 中 `scan_em_dash()` 函数已覆盖。建议每次人工审查前先跑脚本，零误报时确认通过。
14. **Token 膨胀预警**：关系条目较多或较长的角色（如花坂家族成员、晨昏事务所创始人）容易出现 JSON token 超限，审查时重点关注此类角色。
15. **关系 key 称呼规则**：有角色卡的人物使用一般称呼（char_name/char_alias），无角色卡的 NPC（如花坂 枫、花坂 晴子）使用全名。全名 key 出现在花坂家族成员的关系中属正常，不标记为违规。
16. **角色特征差异化**：不同角色的外貌特征不得出现过度相似的描述。包括但不限于：相同的标志性配饰（如皮围裙）、相同的身体习惯描写（如"习惯性眯起眼"）、相同的职业道具描述（如"钢头靴""目镜"）。每个角色应有可区别于其他角色的独立外貌特征。
17. **char_personality 字段格式规范**：core_traits 为 3 个关键词；strengths 为 3 个短词组；flaws 为 3 个短词组；quirks 为 1 个短句。四字段均使用分号分隔。禁止：叙事句、文学修辞、比喻排比、翻译腔句式（"宁可…也不"）、临床术语（"条件反射"）、商业用语（"包装"）、非英语词汇（"Tsundere"）、解释性后缀（"做XX时养成的习惯"）。正例参考四色音四人定稿。

## 第三步：基础格式层审查

### 3a. JSON 语法自动化校验

对角色卡文件夹内所有 `.json` 文件，使用 `python -m json.tool` 逐一解析。解析失败的文件直接标记为 **T0**。

### 3b. JSON 角色卡内容校验

> **枚举值来源约定**：`char_identity` 和 `char_rank` 的合法枚举值以 `角色卡标准格式.md` 和 `评级.md` 为准。
> JSON 中所有字段值已统一使用中文（如 `狩灵` / `游魂` / `罪灵`），与简介保持一致。

对每个 JSON 文件检查：

- [ ] 必填字段完整性（对照 `角色卡标准格式.md` 的字段树）
- [ ] `char_rank` 枚举值合法性（对照 `角色卡标准格式.md`：`1阶`~`5阶` / `色彩阶` / `None` / `传闻级` / `怪谈级` / `梦魇级` / `灾厄级` / `终焉级`）
- [ ] `char_identity` 枚举值合法性（对照 `角色卡标准格式.md`：`狩灵` / `游魂` / `罪灵` / `异常`）
- [ ] `char_special_abilities` 为空时是否已省略整个字段（不允许留空 `{}`）
- [ ] `char_special_abilities` 中：带 `灵力回路 -` 前缀的为灵力回路，无此前缀的为特殊能力（如花坂血统），二者字段归类正确
- [ ] `char_dialogue_examples` 中 `response` 是否使用英文直双引号 `""`
- [ ] `char_dialogue_examples` 中是否出现 ControlBoundary 违规（预设用户行动/想法/回应）
- [ ] `char_dialogue_examples` 中无"声音/语气 XX 得像 XX"句式重复（同质化高频句式，非禁止所有比喻）（`狩灵开场白创作指导.md` §4.4）
- [ ] `char_dialogue_examples` 中三个 response 的动作描写无明显重复，同一动作词（如"微微一笑""叹了口气""皱眉"）在多个示例中出现 ≥2 次应标记（`狩灵角色卡创作指导.md` §2.4.1）
- [ ] `char_dialogue_examples` 的 response 字段中角色全名使用情况，叙述部分应使用代词（她/他），全名仅允许在 situation 中出现（`狩灵角色卡创作指导.md` §2.4.2）
- [ ] `char_relationships` 的 key：有角色卡的人物使用一般称呼（char_name/char_alias），无角色卡的 NPC（如花坂 枫、花坂 晴子）使用全名。专有名词翻译边界，人名、组织名等专有名词保留中文，普通名词使用英文（如 `Patients` 非 `患者`、`Former classmates` 非 `大学同学`、`Field operatives of 狩灵协会2科` 非 `狩灵协会2科外勤队员`）。组织使用中文全称（与 `char_faction` 等字段一致）
- [ ] `char_faction`、`char_status`、`char_relationships` 等字段中术语使用全称：`狩灵协会X科` 不得简写为 `X科` 或 `department X`；`狩灵协会` 不得写为 `Association`
- [ ] 专有名词翻译边界：英文描述字段中仅翻译专有名词本身，连带普通词保留英文（如 `灵能 alloy` 非 `灵能合金`、`灵力 injection` 非 `灵力注入`）。武器分类标签（`灵能重剑`、`灵能盾牌`）视为整体术语
- [ ] JSON 四段式分部预算达标：A 外壳 300-500、B+C 能力+关系 0-500、D 对话 200-400（`狩灵角色卡创作指导.md` §2.3）
88	- [ ] 英文无缩写：英文字段中无 `don't` `can't` `won't` `doesn't` `hasn't` `you're` `it's` 等缩写，全部使用全展开式（`do not` `cannot` `will not` `does not` `has not` `you are` `it is`）。所有格 `'s` 例外。
- [ ] `char_persona.age` 及 `char_background.origin` 中工作/从业年龄 ≥ 18 岁（按现实法律规定）。觉醒年龄不受此限，世界观设定中觉醒集中在青春期，低于 18 岁的觉醒年龄可接受。
- [ ] 非对话字段中无单独 `—`（U+2014 em dash），`灵力回路-` 前缀中的 `-` 为唯一合法连字符。常见违规模式：用 `—` 连接解释说明（改 `:` 或 `. `）、连接从句（改 `, ` 或 `. `）、包裹同位语（直接删除）。参见 `创作者文件/创作文件/狩灵角色卡创作指导.md` §2.2 常见 em dash 替换对照
- [ ] `char_basic_abilities` 每条格式合规：无特殊内容则与 `基础能力等级示例.md` 原文一字不差（中英文均核对）；有特殊内容则纯自定义、不含标准文本片段。背景理由和经验评价不构成特殊内容，详见 `狩灵角色卡创作指导.md` §3.2
- [ ] JSON 与简介同维度基础能力格式一致（同标准或同自定义），不存在一端标准一端自定义
- [ ] 角色外貌特征无跨角色雷同：标志性配饰、身体习惯描写、职业道具描述等不与已有角色重复。新增角色时须逐项对比同阵营/同职业已有角色的 `appearance` 字段

### 3c. MD 简介格式校验

对每个 `*简介.md` 文件检查：

- [ ] 文件以空行开头，不以换行符结尾（`简介标准格式.md` §1-2）
- [ ] 五段式结构完整，大项间仅一个空行，同节子项间无空行（`简介标准格式.md` §3-4）
- [ ] 三、四若无内容，整个大项已删除且后续序号自动补充（`简介标准格式.md` §5 + `狩灵简介创作指导.md` §4.1）
- [ ] 评级为纯中文格式：狩灵=`X阶`/`色彩阶-称号`，游魂=`无`，罪灵/异常=`X级`，无英文无后缀（`狩灵简介创作指导.md` §2.4）
- [ ] 一般称呼符合三条规则（`简介标准格式.md` 一、一般称呼）
- [ ] 特征 = 发型发色 + 瞳色 + 着装（`狩灵简介创作指导.md` §4.2）
- [ ] 评估方署名格式正确（`简介标准格式.md` 五、评估方）
- [ ] 标点合规：无 Markdown 装饰符、无中文引号、无英文逗号出现于中文语境

### 3d. MD 开场白格式校验

对每个 `*开场白.md` 文件检查：

- [ ] token 数在 900-1,100 区间内（使用 tiktoken `cl100k_base` 编码计量）
- [ ] 三段式叙事要素完整（场景建立 → 角色行动 → 标志收尾）
- [ ] 全文第三人称，无预设用户行动（ControlBoundary）
- [ ] 允许模糊预设玩家存在，不明确设定玩家身份/目的/关系（`狩灵开场白创作指导.md` §1.1）
- [ ] 说话内容用英文直双引号 `""` 包裹
- [ ] 叙述部分标点：以 `，。` 为基础，`、` 用于并列枚举，`；` 仅限并列分句内部已含逗号需标层次时，`：` 仅限言语提示动词和字段标签。**特别注意**：非言语提示动词后引出对话必须用逗号 `，"`，不能用冒号 `："`。`——` `……` 仅限对话内，叙述中禁止。对话部分可用 `？！——……`
- [ ] 无 Markdown 装饰符（`*` `_` `~`）
- [ ] 无星号包裹动作（如 `*她笑了笑*`）
- [ ] 无英文 em dash `—`、英文省略号 `...`、中文卷曲引号 `""`
- [ ] 内心独白 ≤ 1 句，用行动暗示
- [ ] 无语境无过渡的无声跳转
- [ ] 动词驱动叙事，无名词堆砌
- [ ] 比喻边界：禁止抽象→抽象（如"平稳得像在陈述事实"）；允许抽象→实物（如"声音像石子"）或实物→抽象（如"装备库安静得像凝固了"）。另检查不同角色是否共用相同罕见音貌比喻（`狩灵开场白创作指导.md` §4.4）
- [ ] 无连续多个问句，开场白中出现 ≥3 个独立发问的句时，标记为"问题循环风险"（`狩灵开场白创作指导.md` §4.4）。连续两个问句可以接受。

### 3e. 泛格式问题检查（适用于所有文本文件）

对所有文件检查：

- [ ] 括号、引号成对闭合（中英文混用场景）
- [ ] 无连续两个以上空行
- [ ] 无段落首尾多余空白字符
- [ ] 无明显错别字或拼音输入法残留

> **自动化辅助**：以下检查项已有脚本支持，建议在人工审查前先跑一遍，零误报时确认通过：
> - JSON 非对话字段 `—`（U+2014）扫描（排除 `灵力回路-` 和对话字段）。**注意**：正则 `—(?!—)` 会误报 `——` 的第二个字符（后跟 `"` 等非 `—` 字符时），需人工确认每个命中是否为 `——` 破折号的尾部。
> - 简介+开场白中文标点合规扫描（禁止项：单独 `—`、`–`、`...`、开场白 `""` 卷曲引号、Markdown 装饰符、`{{char}}`/`{{user}}`）
> - 脚本输出按文件分组，人工仅需确认离群值

### 输出格式

按文件夹分批输出，零问题文件汇总一行，有问题文件逐个展开：

```
【文件名】→ 问题类型 → 具体位置（引号标出原句）→ 严重等级（T0/T1/T2）
```

## 第四步：逻辑一致性层审查

### 4a. 同角色三文件交叉比对

对每个角色，比对 JSON ↔ 简介 ↔ 开场白：

- [ ] 基础信息一致：姓名、性别、年龄、身高、体重、从属、身份、评级
- [ ] 基础能力一致：灵力、灵视、身体素质（等级名与描述对应）
- [ ] 特殊能力一致：回路名称、描述数量不增减
- [ ] 装备信息一致：武器描述在不同文件中不矛盾
- [ ] 性格特质一致：简介中的综合评估与 JSON 中的 core_traits/strengths/flaws 不矛盾
- [ ] 简介中的评级与 JSON 中 char_rank 一致（均为中文）
- [ ] 简介中关系对象使用一般称呼（非全名）

### 4b. 跨角色矛盾检查

- [ ] 同一事件在不同角色卡中的时间、地点、参与者描述是否一致
- [ ] 同一组织/地点在不同角色卡中的描述是否彼此矛盾
- [ ] 角色关系双向一致（A 对 B 的关系描述与 B 对 A 的关系描述是否匹配）。检查要点：双方年龄/辈分逻辑一致（前辈/后辈方向正确）、情感态度吻合（如骄傲←→仰望）、关系起因不矛盾

### 4c. 与世界观设定一致性

- [ ] 角色所属科室与 `狩灵协会.md` 中科室定义一致（地点、职能）
- [ ] 角色评级与 `评级.md` 中评级体系一致
- [ ] 角色身份（狩灵/游魂/罪灵/异常）与 `灵能者.md` 中定义一致
- [ ] 特殊能力中的术语与 `世界书设定/` 一致

### 4d. 示例对话与性格一致性

- [ ] `char_dialogue_examples` 中的语言风格与 `char_personality.core_traits` 匹配
- [ ] response 内容不包含 ControlBoundary 违规
- [ ] 三个 response 的动作描写有明确侧重区分（手部动作 / 面部表情+身体姿态 / 位移动作），避免 AI 输出陷入单一动作模式（`狩灵角色卡创作指导.md` §2.4.1）

### 输出格式

## 第五步：汇总分级 + 修复优先级建议

将所有问题按三级汇总：

| 等级 | 定义 | 示例 |
|------|------|------|
| **T0（必须修复）** | JSON 语法错误、必填字段缺失、char_rank 非法值、评级格式严重错误（可能导致解析失败）、核心设定与世界书冲突、时间线严重错乱、ControlBoundary 严重违规（明确预设用户身份/行动） |

1. 完整问题分级清单（按 T0 → T1 → T2 排列）
